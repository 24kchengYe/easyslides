# 科研学术 PPT 工作流教程

> 本文用于持续沉淀 EasySlides/EasyPPT 的教程型问答：当用户提供论文、数据、讲稿、模板或其他科研材料时，如何把材料规划成一套真正可讲、可编辑、证据可追溯的学术 PPT。

## 如何使用这个文件

这个文件不是一次性说明书，而是一个可以持续追加的教程容器。之后如果有新的判断、比较、工作流原则、用户问答或产品定位，都可以按“教程片段”的形式加入。

建议每个片段保留三层内容：

1. 用户原始问题。
2. 可以直接对外解释的回答。
3. 可以沉淀进系统工作流的原则。

## 教程片段 1：从论文 PDF 规划学术 PPT

### 用户问题

当用户给你一篇或者多篇论文的 PDF，你是如何规划用它们做 PPT 的呢？

### 核心回答

我通常不会把 PDF “摘要成一堆页面”，而是先把它们变成一个**可讲的学术故事**。

关键不是把论文内容压缩进幻灯片，而是完成三次转换：

1. 从 PDF 到**证据库**：提取研究问题、方法、结果、图表、结论和引用位置。
2. 从证据库到**论证结构**：判断哪些信息服务于同一条主线，哪些适合比较、补充或放入附录。
3. 从论证结构到**页面设计**：为每页确定一个主张、一个主要证据和一种合适的视觉表达。

### 规划流程

#### 1. 先锁定汇报场景

正式规划前，先确认这套 PPT 的使用场景，而不是直接读 PDF 开始分页。

需要锁定的信息包括：

| 项目 | 说明 |
|---|---|
| 汇报类型 | 文献汇报、组会、课程展示、会议报告、开题、答辩、基金汇报、多篇综述等 |
| 听众 | 同方向专家、跨学科听众、导师/评委、学生、项目评审等 |
| 时长与页数 | 例如 8 分钟 8 页、20 分钟 20 页、正式答辩 40 页等 |
| 输出语言 | 中文、英文或双语 |
| 视觉风格 | 清爽组会风、正式答辩风、会议报告风、基金蓝色风、本地模板库风格等 |
| 输出形式 | 默认应为可编辑原生 PPTX，而不是整页截图或图片堆叠 |

#### 2. 拆论文，不急着做页

对每篇 PDF，先提取结构化信息：

| 信息类型 | 提取内容 |
|---|---|
| 研究问题 | 论文试图解决什么问题 |
| 研究缺口 | 过去方法或认识哪里不足 |
| 核心贡献 | 作者真正新增了什么 |
| 方法与数据 | 实验设计、模型、样本、数据来源、参数与流程 |
| 关键结果 | 支撑结论的主要发现 |
| 图表资源 | 可用于 PPT 的 figures、tables、equations、schematics |
| 局限性 | 假设、边界条件、未解决问题 |
| 可延展方向 | 对后续研究、项目或课题的启发 |
| 来源定位 | 页码、图号、表号、caption、DOI、引用信息 |

这一步的产物不是 PPT 页面，而是一个可以追溯的 `source_pack` 或证据清单。

#### 3. 单篇论文：找主线

单篇论文不要机械按 Introduction、Methods、Results、Discussion 搬运。

更适合的叙事骨架是：

```text
问题背景 -> 关键缺口 -> 作者思路 -> 方法设计 -> 关键证据 -> 结论贡献 -> 局限与启发
```

每一页最好只承担一个明确主张，例如：

- “该论文的真正创新不在模型更大，而在约束了特征表达。”
- “结果提升主要来自数据组织方式，而不是训练轮数增加。”
- “作者的实验验证了机制假设，但外推到真实场景仍有边界。”

页面标题应尽量写成结论句或判断句，而不是只写“Introduction”“Method”“Result”。

#### 4. 多篇论文：做综合，而不是拼接

多篇论文不能简单做成“论文 1 十页 + 论文 2 十页 + 论文 3 十页”。

应先建立比较矩阵：

| 比较维度 | 需要回答的问题 |
|---|---|
| 研究问题 | 它们是否在解决同一个问题？问题定义是否一致？ |
| 方法路线 | 技术路线、理论假设、实验设计有什么差异？ |
| 数据与尺度 | 数据来源、样本规模、空间/时间尺度是否可比？ |
| 关键结论 | 哪些结论互相支持？哪些结论存在冲突？ |
| 证据强度 | 哪篇证据更直接？哪篇更像相关性或间接推断？ |
| 研究空白 | 多篇论文共同暴露了什么尚未解决的问题？ |

然后把 PPT 组织成：

```text
领域背景 -> 问题演化 -> 方法谱系 -> 关键证据比较 -> 共识与争议 -> 研究空白 -> 对自己工作的启发
```

这样多篇论文会变成一个领域图谱，而不是几篇论文的串讲。

#### 5. 先产出 deck plan

在真正生成 PPTX 前，先写出每页的角色和证据绑定。

一个常见的论文汇报页面结构如下：

| 页面 | 页面角色 | 主要内容 |
|---|---|---|
| 1 | 封面 | 题目、论文信息、汇报人 |
| 2 | 背景与问题 | 为什么这个问题重要 |
| 3 | 研究缺口 | 现有方法或认识的不足 |
| 4 | 核心思路 | 作者提出的解决路径 |
| 5 | 方法框架 | 实验、模型或技术路线 |
| 6 | 关键结果 1 | 最核心的图表或发现 |
| 7 | 关键结果 2 | 补充证据或机制解释 |
| 8 | 对比与讨论 | 与已有方法、其他论文或不同条件比较 |
| 9 | 局限与启发 | 论文边界、潜在问题、后续方向 |
| 10 | 总结 | 一句话贡献、两到三个 takeaways |

每页都应具备三个元素：

1. **一句主张**：这页到底想让听众相信什么。
2. **一个主要证据**：图、表、流程图、公式、机制图或对比矩阵。
3. **一个来源标注**：页码、图号、论文引用或用户明确提供的材料。

#### 6. 再决定视觉与模板

视觉风格应服务于汇报场景：

| 场景 | 推荐风格 |
|---|---|
| 组会/文献汇报 | 信息密度适中，图表优先，标题清楚 |
| 课程展示 | 解释性更强，概念图和过渡页更多 |
| 会议报告 | 一页一个信息点，节奏更快，图表更大 |
| 开题/答辩 | 更正式，章节结构清晰，逻辑链完整 |
| 基金/项目汇报 | 强调必要性、创新性、可行性和计划 |
| 多篇综述 | 强调比较矩阵、时间线、方法谱系和争议地图 |

默认原则是做**可编辑原生 PPTX**：

- 标题、正文、箭头、表格、注释、流程图尽量可编辑。
- 论文原始图表可以作为 source-bound raster assets 保留。
- 不把整页做成截图。
- 不凭空发明论文没有的数据、标签、p 值、机构、结论或引用。

#### 7. 最后做样页和质量检查

对正式任务，应先做 3-5 页样页确认叙事和风格，再扩展全套。

最终检查包括：

| 检查项 | 目标 |
|---|---|
| 叙事检查 | 页面顺序能否自然讲通 |
| 证据检查 | 每个主张是否有来源支撑 |
| 引用检查 | 图表、数据、方法和观点是否有标注 |
| 可编辑性检查 | 文字、形状、表格、箭头等是否可编辑 |
| 版式检查 | 是否有文字溢出、遮挡、错位、过小字体 |
| 图表检查 | 关键图是否清晰，caption 是否准确 |
| 模板检查 | 是否残留占位符、默认文本或无关元素 |

## 教程片段 2：EasySlides 与 PPT Master 的区别

### 用户问题

我们的项目和 PPT Master 的项目区别是什么？除了我设计了适合学术的模板。

### 核心回答

区别不只是“我们有学术模板”。更本质的区别是：

**PPT Master 更像通用 PPT 生产引擎和模板学习系统；EasySlides/EasyPPT 是面向科研学术表达的完整工作流。**

PPT Master 解决“如何稳定生产高质量可编辑 PPT”；EasySlides/EasyPPT 解决“科研材料应该如何被组织成可信的学术表达，并用可编辑 PPT 呈现出来”。

### 对比维度

| 维度 | PPT Master | EasySlides/EasyPPT |
|---|---|---|
| 核心定位 | 通用 PPT 生成、模板复用、SVG 到可编辑 PPTX | 科研材料到学术表达的完整系统 |
| 起点 | 主题、材料、模板、设计需求 | 论文、数据、图表、讲稿、课题材料、科研场景 |
| 第一原则 | 做出可编辑、风格一致、模板可复用的 PPT | 保证学术内容可信、可讲、可追溯，再做成 PPT |
| 模板角色 | 模板是重要输入，负责视觉与结构复用 | 模板只是视觉容器；学术场景先决定内容组织 |
| 内容规划 | 通用 deck plan / design spec | `source_pack`、`figure_index`、`story_blueprint`、claim-evidence-citation |
| 学术约束 | 不是主目标 | 核心目标：不编造结论、保留引用、图表可追溯、证据支撑每页主张 |
| 场景系统 | 更偏模板/页面类型 | 单篇文献、多篇综述、答辩、开题、基金、组会、会议报告、培训、进展汇报等 |
| 质量检查 | SVG/PPTX 可编辑性、版式、导出、预览 | 额外检查学术逻辑、引用、证据、图表可读性、claim 是否成立 |
| 产物目标 | 好看的可编辑 PPT | 能被讲、能被问、能被追溯的学术 PPT |

### 我们吸收 PPT Master 的部分

EasySlides/EasyPPT 并不是否定 PPT Master，而是在它的底层工程能力上增加科研学术编排层。

主要吸收的是：

1. `.pptx` 模板可以被拆成事实包：母版、布局、占位符、颜色、字体、预览图、素材。
2. SVG/shape IR 到 DrawingML 的可编辑导出路线。
3. 模板资产库、页面复用、占位符替换、视觉 QA。
4. `design_spec.md`、`spec_lock.md` 这种把需求锁住再生成的工程纪律。

### 我们新增的学术系统

EasySlides/EasyPPT 在 PPT Master 的工程层之上，增加了一套学术表达系统：

1. **source truth**：先抽取论文、数据、图表和讲稿事实，不直接写页。
2. **evidence governance**：每个主张绑定来源、页码、图号、表号或用户材料。
3. **academic scenario routing**：先判断是什么科研场景，再选模板。
4. **story blueprint**：先确定听众、中心问题、论证路径。
5. **figure/table index**：判断哪些图进主线，哪些进附录，哪些要拆分或转译。
6. **academic QA gate**：检查是不是只有标题好看但证据链不成立。
7. **scenario_variant**：没有现成模板也不硬套答辩/文献汇报，而是记录新的学术场景变体。

### 模板不是根本区别

学术模板当然重要，但它只是可见层。真正的区别是：

```text
PPT Master:
模板/内容需求 -> 设计规范 -> SVG/PPTX -> 视觉与导出检查

EasySlides/EasyPPT:
科研材料 -> source truth -> 证据系统 -> 学术故事 -> 场景路由
-> 模板绑定 -> 可编辑 PPTX -> 学术 QA + 视觉 QA
```

所以可以把关系说成：

**PPT Master 是底层 PPT 工程能力；EasySlides/EasyPPT 是科研学术场景的上层智能编排。**

模板只是表层差异，真正的差异是：EasySlides/EasyPPT 把“学术可信度”和“讲述逻辑”放在 PPT 生成之前。

## 教程片段 3：我们不是照搬 PPT Master

### 用户问题

在技术架构上、或者技能的设计上，我们是照搬 PPT Master 吗？其实我希望在教程里能体现出我们的特色或优势，不然别人会说，那我为什么不用 PPT Master 呢？

### 核心回答

不是照搬。

更准确地说，EasySlides/EasyPPT **吸收了 PPT Master 在可编辑 PPTX、模板事实包、SVG/shape IR、导出检查上的工程能力**，但把整个系统的中心从“通用 PPT 生产”移到了“科研学术表达”。

如果只看底层导出链路，两者确实有亲缘关系；但如果看用户输入、规划中间件、页面决策、质量门禁和 skill 触发逻辑，EasySlides/EasyPPT 已经是不同的系统。

### 技术架构上的差异

| 层级 | PPT Master 的重点 | EasySlides/EasyPPT 的重点 |
|---|---|---|
| 输入层 | 主题、内容、模板、PPTX 参考 | 论文、数据、图表、讲稿、课题材料、科研场景、模板 |
| 解析层 | 把材料和模板转成可生成的设计信息 | 先建立 `source_pack`、`figure_index`、引用和证据索引 |
| 规划层 | `deck_plan` / `design_spec` / `spec_lock` | 先做 audience-state、story blueprint、claim-evidence-citation，再进入 design spec |
| 场景层 | 通用页面类型和模板角色 | 学术场景 profile、`scenario_variant`、科研听众与论证目标 |
| 模板层 | 模板是主要复用资产 | 模板是视觉容器，不能替代学术场景和证据逻辑 |
| 生成层 | SVG/shape 到可编辑 PPTX | 继承可编辑生成，但受学术证据、引用、图表可读性约束 |
| QA 层 | 版式、导出、预览、占位符、可编辑性 | 额外检查学术 claim 是否有证据、图表是否可追溯、引用是否保留 |

也就是说，PPT Master 更像一个强大的 **PPT 工程引擎**；EasySlides/EasyPPT 在这个能力之上加了一层 **科研表达编排层**。

### Skill 设计上的差异

PPT Master 的 skill 更适合回答：

- 如何生成一套 PPT？
- 如何复用一个模板？
- 如何把 SVG 稳定转成可编辑 PPTX？
- 如何检查导出和版式？

EasySlides/EasyPPT 的 skill 还要回答：

- 这是什么学术场景？
- 这份材料里哪些是事实，哪些是推断？
- 哪些图表能支撑主线，哪些只能放附录？
- 每一页的 claim 是否有 evidence？
- 这个模板是否适合当前科研听众和汇报目的？
- 如果没有对应模板，如何保留学术结构而不是硬套答辩或文献汇报？

所以我们的 skill 不是把 PPT Master 的 prompt 换成“学术版”，而是重新设计了上游决策：

```text
PPT Master skill:
需求/模板 -> 设计规范 -> 页面生成 -> PPTX 导出 -> 视觉 QA

EasySlides/EasyPPT skill:
科研材料 -> source truth -> 证据索引 -> 学术场景路由
-> 故事蓝图 -> 页面 claim/evidence/citation
-> 模板绑定 -> 可编辑 PPTX -> 学术 QA + 视觉 QA
```

### 我们没有照搬的地方

1. **没有让模板决定场景**  
   选择了答辩模板，不代表内容就必须是答辩；选择了文献模板，也不代表多篇综述要被拆成几篇论文串讲。场景先行，模板后绑定。

2. **没有把论文当普通内容素材**  
   论文不是一段长文本，而是带有研究问题、方法、数据、图表、引用和证据链的学术对象。系统必须先抽取和验证这些信息。

3. **没有只追求漂亮页面**  
   对科研 PPT 来说，漂亮但证据错误是失败；图表清晰、引用准确、逻辑可讲比装饰更重要。

4. **没有把 PPTX 导出当终点**  
   导出之后还要检查学术逻辑、引用、图表、可编辑性、占位符、文本溢出和视觉预览。

5. **没有把当前模板库当能力边界**  
   现在没有某个学术场景模板，不代表系统不能做这个场景。可以记录 `scenario_variant`，先保证内容组织正确，再选择自由设计、通用学术模板或用户提供模板。

### 为什么不用 PPT Master 就够了

如果用户要做的是通用商业 PPT、品牌介绍、活动宣讲、普通模板复刻，直接用 PPT Master 很合理。

但如果用户要做的是科研学术 PPT，尤其涉及论文、数据、图表、引用、答辩、基金、组会、会议报告、课程讲授、科研训练或多篇综述，只用 PPT Master 容易缺少三件事：

1. **学术内容治理**：哪些 claim 可以讲，依据在哪里，是否过度推断。
2. **科研图表治理**：哪些图要保留、拆分、转译、标注或放附录。
3. **学术场景治理**：不同场景的听众、节奏、证据密度、结尾方式和问答准备不同。

EasySlides/EasyPPT 的优势不是“比 PPT Master 更会做所有 PPT”，而是：

**当任务是科研学术表达时，它更知道什么该讲、凭什么讲、讲给谁听，以及如何把这些约束落到可编辑 PPTX 里。**

### 可以对外这样解释

PPT Master 是通用 PPT 工程底座，EasySlides/EasyPPT 是科研学术表达系统。

我们吸收了 PPT Master 的可编辑 PPTX 工程能力，但没有停在“模板生成器”。我们把论文、数据、图表、引用、听众和学术场景变成了生成前的硬约束，再让模板和导出系统服务这些约束。

因此，EasySlides/EasyPPT 的价值不是“又做了一个 PPT Master”，而是把 PPT Master 擅长的底层生产能力，推进到科研学术这个更高要求的场景里。

## 可复用的一句话原则

把论文 PDF 做成 PPT，不是把 PDF 缩短，而是把论文重构成面向听众的论证。

我会先把论文转成证据库，再转成学术故事，最后才转成页面。这样做出来的 PPT 更像一场真正的学术表达，而不是 PDF 的压缩版。

PPT Master 是底层 PPT 工程能力；EasySlides/EasyPPT 是科研学术场景的上层智能编排。

EasySlides/EasyPPT 不是 PPT Master 的学术皮肤，而是把可编辑 PPTX 工程能力接入科研证据、学术场景和讲述逻辑之后形成的上层系统。
